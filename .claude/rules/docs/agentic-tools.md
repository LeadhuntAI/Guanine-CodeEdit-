## Purpose
Provides a sandboxed filesystem and code search toolkit for the agentic engine. Each tool module exposes a standardized `execute()` function that accepts a `_base_dir` parameter for path sandboxing and returns JSON strings for safe inter-process communication. The `definitions.json` serves as a declarative registry enabling dynamic tool discovery and grouping into logical sets (`code_analysis`, `file_ops`).

## Data Flows
- **Input:** The agentic engine (`tool_executor.py`) dispatches tool calls by importing `execute()` from the function path specified in `definitions.json`, passing named parameters plus `_base_dir` for sandboxing.
- **Processing:** Each `execute()` validates the path via `_validate_path()` (resolves symlinks, checks prefix containment), performs the filesystem operation, and catches all exceptions into JSON error responses.
- **Output:** Every tool returns a JSON string — either a result payload (`content`, `entries`, `matches`, `tree`, `written`) or an `{"error": "..."}` object.

## Invariants Enforced
- **Path sandboxing:** All tools resolve paths via `os.path.realpath` and reject any path that escapes `_base_dir`. This prevents directory traversal attacks (e.g., `../../etc/passwd`).
- **JSON contract:** Every `execute()` function returns a `str` containing valid JSON — never raises exceptions to the caller.
- **Recursive depth limits:** `list_directory` caps recursion at depth 3; `get_file_tree` defaults to depth 4.
- **Result truncation:** `search_code` stops after 100 matches (`_MAX_MATCHES`) and sets `"truncated": true`.
- **Skip lists:** `search_code` and `get_file_tree` skip common build/cache directories (`.git`, `node_modules`, `__pycache__`, `.venv`, etc.) and binary extensions (`.pyc`, `.so`, `.dll`, etc.).

## Shared Types

### Tool `execute()` interface (defined here, used by agentic-engine)
Standardized signature: `execute(named_params, _base_dir='.', **kwargs) -> str` where the return value is always a JSON string. This is the contract between tool modules and the engine's `tool_executor.py`.

### Tool definitions schema (defined here, used by agentic-engine, agentic-docs-and-workflows)
`definitions.json` declares each tool's `name`, `description`, `function_path`, and `parameters_schema` (JSON Schema). The `tool_sets` key groups tools into `code_analysis` and `file_ops` sets for workflow composition.

## Configuration
- **`definitions.json`** — Declarative tool registry. Maps tool names to `function_path` strings (e.g., `agentic.tools.read_file.execute`) and defines parameter schemas. The engine reads this to discover available tools.
- **`_base_dir` parameter** — Runtime sandbox root, passed by the engine to constrain all filesystem operations. Not a config file but a runtime injection.

## Impact Notes
- **`agentic-engine`** is the primary consumer. Changing any `execute()` signature, return format, or `function_path` in `definitions.json` will break tool dispatch in `tool_executor.py`, `runner.py`, and `loop.py`.
- **`agentic-docs-and-workflows`** references tool names and schemas. Renaming tools or changing parameter schemas requires updating documentation.
- The `_validate_path` function is duplicated across four modules (`read_file`, `write_file`, `list_directory`, `search_code`). A bug fix or behavioral change must be applied to all four copies.
- Adding a new tool requires: (1) creating the module with an `execute()` function, (2) adding its entry to `definitions.json`, and (3) optionally adding it to a `tool_sets` group.
