# Code Documenter — Analyse, Document, and Maintain

This skill has two modes:

- **CREATE mode:** Analyse a file or module from scratch, generate a documentation rule, and register it in the index.
- **UPDATE mode:** After code changes, refresh an existing documentation rule to stay in sync.

The user triggers CREATE with phrases like "document this", "create a rule about this file", "analyse this and save it".
The user triggers UPDATE with phrases like "update the rule", "sync the docs", "we just changed this — update it".

---

## Phase 1: Determine Mode and Scope

1. **Identify the target.** What file or module is the user referring to?
   - If they said "this file" — check which file was just open or edited in the conversation.
   - If they named a path — use it directly.
   - If they said "what we just worked on" — scan the conversation history for the most recently edited files.

2. **Determine the mode:**
   - No existing rule for this file → **CREATE mode** (go to Phase 2)
   - Existing rule found → **UPDATE mode** (go to Phase 6)

3. **Check for an existing rule.** Search for it:
   ```
   grep -r "<filename or module name>" .claude/RULES_INDEX.md */.claude/RULES_INDEX.md .claude/rules/ */.claude/rules/ 2>/dev/null
   ```

4. **Identify which module/app this belongs to.** From the file path, determine
   the module or app. This determines where the doc rule and index entry go.

**Validation Gate:** Confirm the target file exists and you know the module. State the mode (CREATE or UPDATE) before proceeding.

---

## Phase 2: Deep Analysis (CREATE mode)

Read the target file fully. Then systematically extract the following.

### 2a. Structural Overview

- **File type:** Identify what role this file plays (model, controller, service, utility, configuration, worker, test, etc.)
- **Patterns used:** List framework-specific patterns and language idioms
- **Imports:** List all imports. Group into: standard library, third-party, internal (other project modules)

### 2b. Definitions Inventory

For each class, function, or constant defined in the file, extract:

| Element | What to capture |
|---------|----------------|
| **Classes** | Name, base classes, key methods, properties, purpose |
| **Functions** | Name, parameters (with types if available), return type, purpose |
| **Endpoints/Routes** | Method, path, request/response types, auth requirements |
| **Constants/Config** | Name, value/default, where used |
| **Types/Interfaces** | Name, fields, where used |

### 2c. Dependency Mapping

Trace how this file connects to the rest of the system:

- **Inbound dependencies** — who imports or calls things from this file?
  ```
  grep -r "from.*<module> import\|import.*<module>" --include="*.<ext>" -l
  ```
- **Outbound dependencies** — what does this file import from other project modules?
- **Data flow participation** — which critical flows (from CLAUDE.md) does this module participate in?
- **Invariants enforced** — which design invariants (from `.claude/rules/invariants.md`) does this module help enforce?

### 2d. Impact Analysis

For each major element, write one sentence describing what would break or change if this element were modified. Focus on:

- Which other modules import this directly
- Which API endpoints expose this
- Which data flows would be affected
- Which invariants could be violated

**Validation Gate:** Present the analysis summary to the user. Ask: "Does this look complete, or should I dig deeper into any area?" Do not generate the rule file until confirmed.

---

## Phase 3: Generate the Documentation Rule

Create a markdown rule file that captures everything from Phase 2.

### File location and naming convention

Documentation rules live inside the module's `.claude/rules/docs/` folder:

```
<module>/.claude/rules/docs/<topic>.md
```

For cross-module documentation, use the project root:
```
.claude/rules/docs/<topic>.md
```

### Required structure of the generated rule

```markdown
# [Project] — [Module Name] Documentation

> Auto-generated by code-documenter. Last updated: <YYYY-MM-DD>
> Source: `<full file path>`

## Purpose

<2-3 sentences: what this module does and why it exists>

## Data Flows

<Which critical flows this module participates in>

## Invariants Enforced

<Which design invariants this module helps enforce, with brief explanation>

## Definitions

### Functions / Classes

#### `function_name` or `ClassName`

- **Type:** Service | Model | Endpoint | etc.
- **Purpose:** <one sentence>
- **Signature:** `def func(param: Type) -> ReturnType`
- **Depends on:** <what it imports or calls from other modules>
- **Used by:** <which files import or call this — file paths>

<repeat for each definition>

## Module Dependencies

| Direction | Module | What | How |
|-----------|--------|------|-----|
| → uses | [module] | [function] | [how it's called] |
| ← used by | [module] | [function] | [how it's called] |

## Configuration

<What this module reads from config/env. List specific fields.>

## Impact Notes

<What to be careful about when changing this file.
Which invariants are at risk. Which data flows would be affected.>

## Change History

| Date | Summary |
|------|---------|
| <YYYY-MM-DD> | Initial documentation generated |
```

### Writing guidelines

- Keep descriptions factual and terse. No filler.
- For "Used by" entries, list file paths not vague descriptions.
- Cap the file at 200 lines. Summarise minor elements in a table.
- Do NOT paste source code into the rule. Describe structure, don't replicate it.
- Always note which invariants the module enforces — this is the most important context for safe editing.

**Validation Gate:** Write the file, then re-read it and verify every cross-reference file path actually exists.

## Phase 4: Register in RULES_INDEX.md

Find the `RULES_INDEX.md` for the relevant module:

```
<module>/.claude/RULES_INDEX.md
```

If cross-cutting, use the root `.claude/RULES_INDEX.md`.

Add or update the entry. Each entry is a markdown link with a one-line summary:

```markdown
- [rules/docs/<module>.md](rules/docs/<module>.md) — <one-line summary>
```

Do not add duplicates. If an entry exists, update its summary.

**Validation Gate:** Read the RULES_INDEX.md back and confirm no duplicate entries exist.

---

## Phase 5: Report to User (CREATE mode complete)

```
Documentation rule created
   File: <path>
   Registered in: <RULES_INDEX.md path>

   Summary: <2 sentences of what was documented>
   Invariants covered: <list which invariants this module enforces>
```

## Phase 6: Update Existing Rule (UPDATE mode)

### 6a. Find the existing rule

```
grep -rl "<filename>" .claude/rules/docs/ */.claude/rules/docs/ 2>/dev/null
```

If not found, switch to CREATE mode (Phase 2).

### 6b. Re-analyse the source file

Re-read the source file. Compare against the existing rule:

1. **New definitions** — functions, classes, endpoints added
2. **Removed definitions** — things documented that no longer exist
3. **Changed signatures** — parameters, return types that changed
4. **Changed dependencies** — new/removed imports
5. **Changed data flow participation** — new flow involvement
6. **Invariant changes** — new/removed invariant enforcement

### 6c. Apply the delta

Edit the existing rule file in place. Do NOT rewrite from scratch. This keeps diffs clean.

- Add new definitions, remove deleted ones
- Update signatures, dependencies, "Used by" references
- Update "Change History" table with today's date
- Update the `Last updated` date in the header

### 6d. Update RULES_INDEX.md if needed

If the summary or scope changed, update the corresponding entry.

### 6e. Report

```
Documentation rule updated
   File: <path>
   Changes:
      - Added: <new elements>
      - Removed: <deleted elements>
      - Updated: <changed elements>
   RULES_INDEX.md: <updated / no change needed>
```

---

## Phase 7: Batch Mode (optional)

If the user says "document all modules" or "create rules for everything":

1. List all source files in scope (excluding tests, configs, init files).
2. For each file, run Phase 2–4.
3. Review the RULES_INDEX.md for completeness.
4. Report a summary.

Limit: Process a maximum of 10 files in batch. If more exist, ask the user to prioritise.

---

## Edge Cases

- **Init files / barrel exports:** Skip unless they contain significant logic.
- **Test files:** Document only if the user explicitly asks.
- **Config files (JSON, YAML, TOML):** Document at the project root level, not in a module's docs.
- **Very large files (500+ lines):** Cap the rule at 200 lines. Summarise minor elements in a table. Prioritise data flows and invariant enforcement.
- **Cross-module functions:** If a function in one module is critical to understanding another, mention it in "Depends on" but don't duplicate its full documentation.
